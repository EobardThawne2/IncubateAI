Since you are racing against the clock for the Promptathon, this implementation plan is optimized for **speed and high-impact visual delivery**.

We are skipping auth and databases for the V1 MVP because they introduce debugging overhead that distracts from the core judging criteria: orchestration and prompt engineering.

Here is your exact execution roadmap to build the **Multi-Agent Business Orchestrator**.

## Implementation Phases

1. **Phase 1: Environment & Repository Setup:** Goal: Boilerplate running.
**1. Backend (Python/FastAPI):**

* Create a virtual environment (`python -m venv venv` or use `uv`).
* Install dependencies: `crewai[tools]`, `fastapi`, `uvicorn`, `langchain-groq`, `pydantic`.
* Create a `.env` file with your `GROQ_API_KEY`.
* **Deliverable:** A `main.py` file with a basic `/health` endpoint running on `localhost:8000`.

**2. Frontend (Next.js):**

* Run `npx create-next-app@latest` (use TypeScript, Tailwind, App Router).
* Initialize `shadcn/ui` with dark mode defaults.
* **Deliverable:** A default Next.js page running on `localhost:3000`.


2. **Phase 2: Backend Core — Pydantic & API Route:** Goal: Data contracts locked.
* Define the exact `Pydantic` schemas for `GenerationRequest`, `OrchestratorPlan`, `CopyAssets`, and `BusinessEngineResponse`.
* Create the `POST /api/generate-business` endpoint in FastAPI.
* Mock the LLM response first. Have the endpoint return a hardcoded JSON payload matching the `BusinessEngineResponse` schema.
* **Deliverable:** The FastAPI endpoint successfully returning the mock JSON when hit via Postman or cURL.


3. **Phase 3: Frontend Core UI & State Machine:** Goal: The visual shell.
* Build the main `app/page.tsx` layout with the sidebar and main canvas.
* Implement the `useState` machine: `idle` | `processing` | `success` | `error`.
* Build the Launchpad view (textarea and submit button).
* Wire the submit button to call your local FastAPI backend (which is currently returning mock data).
* Build the Split-Pane Workspace view to render the mock data into the Markdown, Copy, and Code tabs.
* **Deliverable:** A fully interactive UI that transitions from Launchpad to Workspace using the mock backend data.


4. **Phase 4: The Agentic Engine (CrewAI Integration):** Goal: The brain works.
* In your Python backend, replace the mock data with the actual CrewAI logic.
* Initialize the 4 agents using `ChatGroq` with the specific models (`llama-3.3-70b` for logic/code, `llama-4-scout` for research, `llama-3.1-8b` for copy).
* Write the system prompts for the Orchestrator, Researcher, Content Creator, and Developer.
* Integrate the DuckDuckGo search tool for the Researcher.
* Force the agents to output data matching your Pydantic schemas using CrewAI's `output_pydantic` or `output_json` parameters.
* **Deliverable:** A functioning backend that takes a real prompt, runs the multi-agent cascade, and returns real LLM-generated JSON.


5. **Phase 5: The Sandbox Iframe & Loading States:** Goal: UX Polish.
* **The Iframe:** In the Next.js Workspace view, ensure the `landing_page_html` string is safely injected into the `<iframe>` using the `srcDoc` attribute.
* **The Loader:** Build the Orchestration Tracker UI for the `processing` state. Simulate the agent progression (e.g., Row 1 active, then Row 2) to keep the user engaged during the 60-second wait time.
* **Deliverable:** The user sees a polished loading sequence, and the final HTML generated by the Dev Agent renders perfectly in the mock browser window.


6. **Phase 6: Deployment & Final Polish:** Goal: Ship it.
* **Backend:** Push the Python code to GitHub. Deploy the FastAPI app via Render (Web Service) or Railway. Add your `GROQ_API_KEY` to the platform's environment variables.
* **Frontend:** Push the Next.js code to GitHub. Deploy via Vercel. Add the deployed FastAPI URL to Vercel as `NEXT_PUBLIC_API_URL`.
* Test the live URL end-to-end to ensure CORS headers are correctly configured.
* **Deliverable:** A live, public URL ready for the Promptathon judges.
