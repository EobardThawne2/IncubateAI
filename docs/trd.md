# Technical Requirements Document (TRD)

**Project:** Multi-Agent Business Orchestrator

**Phase:** V1 MVP (Promptathon Build)

**Architecture Style:** Decoupled Client-Server (Stateless)

---

## 1. System Architecture

The system operates on a decoupled architecture, separating the client-side presentation layer from the heavy, long-running Python execution environment. Because multi-agent workflows can take up to 90 seconds, the architecture relies on asynchronous processing and sandboxed rendering.

### Workflow Execution Flow

1. **Client Request:** User submits a prompt via the Next.js UI.
2. **API Handshake:** Next.js sends a `POST` request to the FastAPI backend.
3. **Agentic Pipeline (CrewAI):**
* Orchestrator parses the prompt into a JSON mission payload.
* Research Agent executes live queries via DuckDuckGo.
* Content Agent drafts assets based on the research.
* Development Agent generates the raw UI code.


4. **Data Aggregation:** FastAPI compiles the outputs into a standardized JSON response.
5. **Client Rendering:** Next.js updates the dashboard state and dynamically injects the generated code into a sandboxed `<iframe>`.

## 3. Database & Authentication

* **Database:** **None.** To maximize speed and avoid connection pooling issues during the hackathon, V1 is completely stateless. The frontend React state holds the session data. If a user refreshes, the data clears.
* **Authentication:** **None.** The platform is open-access for the MVP. Securing the underlying LLM API keys via environment variables is sufficient for this phase.

## 4. API Endpoints

The FastAPI backend exposes a single, asynchronous endpoint to handle the orchestration.

## 5. Security Requirements

| Risk Area | Mitigation Strategy |
| :--- | :--- |
| **XSS (Cross-Site Scripting)** | The HTML generated by the Development Agent must *never* be rendered using `dangerouslySetInnerHTML`. It must be injected into an `<iframe>` using the `srcDoc` attribute to isolate script execution from the main application state. |
| **API Key Exposure** | Groq API keys must reside strictly in the FastAPI `.env` file and never be exposed to the Next.js client layer. |
| **CORS (Cross-Origin Resource Sharing)** | FastAPI must be configured with a strict `CORSMiddleware` policy, allowing requests only from the Next.js frontend domain (e.g., `localhost:3000` or the Vercel production URL). |
| **LLM Hallucination/Injection** | Orchestrator prompts must enforce strict JSON output formatting. Pydantic models in FastAPI will validate the LLM output before sending it to the client. |

## 6. Deployment Plan

*   **Frontend Deployment (Vercel):** The Next.js application will be deployed serverless via Vercel for zero-config CI/CD and edge caching of static assets.
*   **Backend Deployment (Render or Railway):** The FastAPI Python server will be deployed as a Web Service. A `Dockerfile` will be included in the repository to guarantee the environment runs identically to the local WSL setup.
*   **Environment Variables:** 
    *   Vercel: `NEXT_PUBLIC_API_URL` pointing to the deployed FastAPI service.
    *   Backend: `GROQ_API_KEY` and `FRONTEND_URL` (for CORS).

---

## 7. Technical Decisions & Rationale

1.  **FastAPI over Node.js/Express:** CrewAI is a Python-native library. Trying to run complex agent workflows in JavaScript/TypeScript currently lacks the mature ecosystem Python offers. FastAPI natively supports asynchronous execution, preventing the backend from blocking while waiting for 90-second Groq responses.
2.  **Model Load-Balancing via Groq:** Using a single LLM for all agents triggers rate limits. We distribute the load: `llama-3.3-70b` for logic/code (heavy reasoning), `llama-4-scout` for research (massive context window), and `llama-3.1-8b` for copy (speed).
3.  **Stateless Execution:** Bypassing PostgreSQL/SQLite eliminates database schema design, migrations, and ORM debugging from the critical path, ensuring the core multi-agent routing is fully operational before the hackathon deadline.
